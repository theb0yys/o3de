# FOA-SDK Research

`Research/` holds bounded investigation, source registers, claim registers, compatibility observations, and
experiment designs that may inform later FOA-SDK decisions.

Research is not an implementation plan, an accepted architecture decision, a compatibility declaration, a
provider qualification, an execution permit, or runtime authority. Nothing recorded here authorises process
launch, plugin execution, game discovery, proprietary-content inspection, build, deployment, game launch,
runtime mutation, save access, or evidence promotion. A normative design may independently authorise a bounded
unit such as the current Core-only external-tool Gate 0; that authority comes from the normative record and is
not created or enlarged by its research sources.

Normative project architecture and delivery ordering remain in `docs/tainted-grail-sdk/`, `ROADMAP.md`, and
accepted code contracts. When a research statement conflicts with those sources, the normative repository
source controls until a separately reviewed change is accepted.

## Research record rules

1. Preserve supplied source reports unchanged below a topic's `inputs/` directory.
2. Replace conversation-local citation tokens with durable source-register entries before relying on a claim.
3. Bind repository observations to an exact path and full commit ID.
4. Bind external observations to a direct source URL, publisher, version or revision where available, and
   retrieval date.
5. Keep facts, inferences, proposals, contradictions, and unknowns distinguishable.
6. Treat tool versions, licences, support status, and game/runtime details as version-sensitive.
7. Record experiments as plans until separately authorised evidence exists.
8. Route any implementation candidate back through the normative design, review, testing, and delivery gates.
