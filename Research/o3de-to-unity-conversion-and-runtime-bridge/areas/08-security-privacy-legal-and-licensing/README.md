# Security, Privacy, Legal, and Licensing

## Trust distinctions

The program keeps these states independent:

- configured or located path;
- inspected component;
- compatible version;
- qualified provider/extension;
- explicitly enabled operation;
- permitted execution;
- accepted result evidence;
- runtime permission.

A hash or signature observation does not by itself prove authenticity, safety, licensing, compatibility, or
runtime authority.

## Security research topics

- path containment, traversal, archive, symlink, and Windows case-collision handling;
- Windows argument encoding, working directories, environment construction, and secret redaction;
- process timeout, cancellation, process-tree cleanup, crash, shutdown, and excess-output behavior;
- provider/project concurrency and stale-lock recovery;
- imported asset, package, importer, and postprocessor code-execution risk;
- raw local logs versus redacted promotable evidence;
- result replay, spoofing, attempt identity, and protected staging;
- technically enforceable network policy rather than an unsupported isolation claim.

## Content and privacy boundary

Do not commit game executables, assemblies, bundles, assets, ripped media, decompiled classes, private paths,
credentials, unredacted private logs, or extracted proprietary data. Synthetic fixtures and original project
content need provenance and licence records. Local-path hashes are still path-derived observations and should
not be treated as safe public identifiers.

## Licensing boundary

Every provider, native extension, package, tool, fixture, and output requires exact source, version, licence,
notices, modification, and redistribution review. The supplied report does not settle Unity plan terms,
unattended/shared-machine automation rights, BepInEx redistribution, or generated runtime-package rights.
Project policy is not legal advice.

Nothing in this area grants execution, redistribution, or publication authority.
