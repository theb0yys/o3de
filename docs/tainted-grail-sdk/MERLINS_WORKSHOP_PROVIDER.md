# Optional Merlin's Workshop Acquisition Provider

## Status

Implemented as the optional `Plugins/Integrations/MerlinsWorkshop` package.
It is an editor-side acquisition qualification provider, not a Unity host,
mod baker, deployment tool or runtime adapter.

## Official source pin

The provider binds the official public toolkit to:

```text
repository: AR-Questline/merlin-workshop
release: 1.1.0
tag: v1.1.0
commit: 073bdab3e09d6adad5003339fc49b021738d71e6
Unity editor: 6000.0.60f1
Unity revision: 61dfb374e36f
Addressables: 2.7.4
```

The official README describes Merlin's Workshop as the modding toolkit for
Tainted Grail: The Fall of Avalon and warns that GitHub's autogenerated ZIP
does not produce a working toolkit. The official project also uses Git LFS
for scenes, images, audio, 3D assets, PDFs, DLLs, archives and other large
content.

The acquisition route is therefore an exact **Git LFS** work order. The
release-archive route remains disabled until an exact official asset name,
size and digest receive a separate review.

## Source and licence register

The committed source register pins six small identity anchors by Git blob
SHA-1:

- `.gitattributes`;
- `.gitconfig.txt`;
- `Assets/CodeMod/ModInfo.cs`;
- `Packages/manifest.json`;
- `ProjectSettings/ProjectVersion.txt`;
- `README.md`.

The English and Polish official licences are present upstream as Git LFS
PDF objects. The provider records both pointer blob identities, LFS SHA-256
values and byte counts. It does not copy either PDF into FOA-SDK and does
not interpret the bespoke licence as an open-source redistribution grant.

A local checkout cannot qualify while either licence remains an LFS pointer.
After both exact PDFs are materialized, the operator must make an explicit
**operator licence acceptance** declaration. Acceptance is not preselected
or inferred from cloning.

## Deterministic work order

`plan` generates canonical JSON with the exact repository and commit,
reviewed tool prerequisites and four inert steps:

1. clone the official repository without checkout;
2. checkout the exact commit in detached-HEAD form;
3. configure Git LFS locally;
4. materialize the exact LFS content.

Every step has `execution_allowed: false`. There is **no process execution**
or network execution implementation in this package. The current
`ExternalToolchain` host remains discovery-only and the integration does
not bypass it.

GitHub's autogenerated ZIP, branch resolution, tag resolution, latest
release lookup and unreviewed release assets are rejected.

## Local checkout qualification

`inspect-local` performs bounded filesystem-only checks:

- a standalone non-symlinked checkout is required;
- `.git/HEAD` must contain the exact commit, not a symbolic branch;
- all source anchors must match their exact Git blob SHA-1;
- both licences must match their exact LFS SHA-256 and byte count;
- materialized licences must have a PDF signature;
- no private checkout path is written into the receipt.

A materialized checkout first receives
`ready_for_operator_license_acceptance`. Only `accept-license` can produce
`exact_checkout_qualified`.

The resulting receipt is eligible to support a **candidate observation**
through `GameInformationAcquisition`. It does not itself create, register,
promote or approve evidence. The active profile, source, chronology and
ExtensionAPI capability gates remain mandatory.

## Authority boundary

The provider permanently grants no:

- clone, download, archive extraction or Unity launch authority;
- catalog or source-registry mutation;
- automatic evidence promotion;
- game launch, deployment or save mutation;
- signing, upload or publication;
- runtime execution.

The official toolkit's own Unity build menu can create Addressables mod
files in the user's game mod directory. This FOA-SDK provider never invokes
that menu or mirrors its behavior.

## Runtime separation

Merlin's Workshop remains an authoring/acquisition integration only.
**Mono** and **IL2CPP** remain separate later packages under
`Plugins/RuntimeAdapters`; they are not combined with this provider and no
runtime code is imported here.

## Validation

Repository validation proves:

- exact official repository, release, commit and toolchain bindings;
- exact anchor and licence LFS identities;
- deterministic non-executable work-order bytes;
- detached-HEAD and symlink refusal;
- LFS-pointer blocking;
- explicit licence acceptance;
- canonical receipt verification and path privacy;
- C++ exact-install-bound provider registration;
- candidate-observation acceptance only at the exact source revision;
- rejection of binary payloads, process code, authority escalation and
  validation-gate removal.

O3DE compilation and compiled CTest remain required because this unit
changes the production-linked C++ provider registry and tests. No Windows
Unity, mod-bake, Mono or IL2CPP result is implied.
