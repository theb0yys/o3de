# Modding Reference

This lane will provide stable navigation for facts and procedures that cut across tutorials and game systems.

## Planned indexes

- glossary and terminology;
- exact game/runtime/loader/framework profiles;
- plugin, package, record, hook, source, evidence, and adapter identity rules;
- compatibility and version history;
- assembly/type/member target notation;
- configuration and persistence conventions;
- logging and diagnostic vocabulary;
- common failure messages and troubleshooting paths;
- dependency, load-order, and patch-collision records;
- package layout and file-placement rules;
- accessibility, UI, security, privacy, legal, and content requirements;
- deprecated hooks, superseded profiles, and migration notices.

## Troubleshooting record shape

Each troubleshooting entry should state:

1. symptom;
2. exact affected profile;
3. likely causes ordered by evidence;
4. safe checks;
5. expected log or validation result;
6. correction;
7. cleanup or rollback;
8. when to stop and collect diagnostics;
9. related hook, system, package, and known-issue records.

## Reference authority

Reference pages may summarize canonical data for humans, but exact schemas, IDs, relationships, evidence, fingerprints, permissions, and package manifests remain owned by their existing FOA-SDK systems. When a rendered summary disagrees with a canonical record, the canonical record wins and the documentation is stale.
